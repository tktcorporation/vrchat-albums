# ハーネスエンジニアリング運用ルール

## 基本理念

「正しいコードを書く」から「エージェントが正しいコードを確実に出力する環境を設計する」へ。

## フィードバック速度の階層

| 層           | タイミング | 仕組み                                      | 対象                       |
| ------------ | ---------- | ------------------------------------------- | -------------------------- |
| PostToolUse  | ミリ秒     | `.claude/hooks/post-edit-lint.sh`           | フォーマット、静的解析     |
| PreToolUse   | ミリ秒     | `.claude/hooks/pre-bash-guard.sh`           | 危険コマンドのブロック     |
| タスク完了時 | 秒         | `pnpm lint:fix` → `pnpm lint` → `pnpm test` | 全プロジェクトのlint/test  |
| CI           | 分         | `.github/workflows/lint-test-cross.yml`     | クロスプラットフォーム検証 |
| Human Review | 時間       | PR レビュー                                 | 設計判断、ビジネスロジック |

**原則: フィードバックは最速の層で返す。CI で初めて検出される違反は、フックで防げないか検討する。**

## 禁止事項と ADR の結合

CLAUDE.md の全禁止事項は ADR 番号を持つ。新しい禁止事項を追加する際は:

1. `docs/adr/` に ADR を作成（コンテキスト、決定、根拠）
2. CLAUDE.md の禁止リストに ADR 番号付きで追加
3. 可能であれば Linter / Hook で機械的に強制

```
禁止事項 ──参照──→ ADR (根拠)
    │                  │
    └──強制──→ Hook / Linter (機械的検証)
```

## ルール追加の判断基準

### Hook で強制すべきもの

- 破壊的操作（revert, reset, force push）
- 実行順序の制約（PR self-review）
- セキュリティ境界（secrets, .env）

### Linter で強制すべきもの

- コードパターン（ts-pattern 使用、Effect TS 使用）
- インポート規約（Electron lazy import）
- 命名規約（ValueObject パターン）

### ルールファイルで記述すべきもの

- 設計原則（堅牢性、UI/UX）
- ワークフロー（CI、テスト）
- アーキテクチャガイド（ログ同期、タイムゾーン）

## 説明的ドキュメントの扱い

記事原則: 「エージェントにとってコードが真実、ドキュメントはオプションの仮定」

- `docs/` 内の説明的ドキュメントは**参考資料**として存在を許容
- CLAUDE.md やルールファイルに説明的内容を複製**しない**
- ADR は**判断記録**であり説明ドキュメントではない（不変）

## Linter 改善サイクル

エージェントの失敗パターンを検出したら:

1. `issues/` にチケットを作成（日付-説明.md）
2. `scripts/lint-*.ts` または `rules/ast-grep/*.yml` で実装
3. `pnpm lint:custom` に統合
4. ルールファイルの該当箇所に Linter 参照を追加
